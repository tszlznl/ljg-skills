# Reading Guide

这份指南负责让外行先发现自己还没有哪些承重概念，再让这些概念在关系和案例中长出来，最后把论文压成一个能解释现象的模型。读论文时完整执行；不要按论文的 Abstract、Method、Experiments 顺序复述。

## 完成标准

读者半年后只需记得三句话：

1. 这篇论文到底被什么问题逼出来。
2. 作者比原来多看见了什么。
3. 这个新看法会改变我的哪个判断或动作。

三句话必须有因果关系。若它们只是三个亮点，说明还没抓住内核。

但三句话只是记忆索引，不是完整理解。读完正文，读者还应能：

- 指出自己原先缺少的全部承重概念；
- 用关系和案例解释这些概念，而不是只背定义；
- 运行一个模型解释原现象，并预判一个关键条件变化后的不同。

## 第一步：判贡献类型

只选一个主类型。混合论文选最能解释「为什么值得写这篇」的那个。

| 类型 | 它真正要改变什么 | 核心问题 |
|---|---|---|
| 解释 / 发现 | 对一个现象的解释 | 原来解释不了什么？作者看见了哪种关系或机制？ |
| 方法 / 干预 | 做事的杠杆 | 改动什么，结果为何会变？ |
| 测量 / 评测 | 判断好坏的尺子 | 旧指标漏掉什么？新尺子让什么变得可见？ |
| 资源 / 系统 | 可做事情的边界 | 原来什么太贵、太慢或做不到？它让什么成为可能？ |
| 综述 / 理论 | 组织事实的框架 | 哪些零散事实被重新放进了同一张图？ |

类型决定洞见的分量。评测论文通常改变「怎么量」，不能直接拔高成「世界如何运作」；资源论文通常扩大可做范围，不一定提供新解释。

## 第二步：选一个具体锚点

找一个外行能看见的困惑、失败或反常结果，从问题一直用到洞见。这个锚点回答「为什么需要这篇」。

好锚点：

- 系统写出了漂亮 proposal，却接错父论文的核心机制。
- 模型答案正确，但推理长度翻了三倍。
- 两个都叫「推理能力」的 benchmark，让模型排名完全相反。

坏锚点是领域背景：「随着人工智能快速发展」「某领域长期面临挑战」。背景没有张力，也无法检验作者是否真的解决了问题。

## 第三步：写暂定命题脊柱

先完成一句工作句：

> 这篇面对的是「{具体问题}」；作者真正看见的是「{新关系、机制或尺度}」；所以我以后会「{新的判断或动作}」。

检查：

1. 中间若只能填论文名、方法名或 benchmark 名，继续读。
2. 三个分句若没有因果关系，重新选主命题。
3. 最后若只是「值得关注」「重新思考」，把它改成一个问题或动作。

这句工作句不必原样放入输出。此时它只是扫描仪：用来暴露哪些词承担了理解，却还没有真正成为读者的概念。完成第四、五步后必须重写一次。

## 第四步：建立概念缺口账本

沿着下面这条承重链扫描：

```text
具体现象 -> 论文的新判断 -> 使判断成立的关系 -> 证据怎样观察这条关系
```

对链上的每个承重词，不问「我见过吗」，而问：

1. 我能用普通话说出它区分了什么，以及它不是什么吗？
2. 我知道它依赖什么、影响什么，和相邻概念是什么关系吗？
3. 我能在论文案例中指出它出现在哪里，并识别一个不属于它的案例吗？

任一项答不出，就记入内部账本。特别警惕「假熟悉」：能翻译术语、背出定义、看懂公式符号，却不能完成上述判断。条件变化的预测留到第六步，在概念已经接成模型后检查。

账本只收承重概念：拿掉它，中心命题、作用关系或证据解释就会断裂。论文里的工具名、数据集名、实现参数和外围术语不是因为陌生就自动进入账本。对真正承重的缺口要全部记录，不因追求短而跳过。

## 第五步：让概念在关系和案例中长出来

按依赖顺序处理账本，不按术语在论文中的出现顺序排列。先解释后续概念依赖的基础区别，再解释论文新增、重命名或重新连接的概念。

每个概念至少完成三件事：

- *区别*：用人话说明这里具体指什么、不指什么，并给出一个可观察标志；
- *关系*：说明它在什么条件下影响谁、依赖谁，先引起什么变化；
- *案例*：回到贯穿全文的具体锚点，指出哪个动作或结果体现了这条关系；必要时补一个反例或边界案例来完成区分。

不要写孤立词典。一个概念的释义之后若不能立刻接上关系和案例，它仍只是一个词。类比只有在角色、作用方向和结果都能映射时才有用。

全部承重概念补齐后，重写中心命题。若新命题仍含有无法完成上述三项检查的承重词，回到概念账本继续补。

## 第六步：把概念接成解释模型

模型的最小骨架是：

```text
条件/输入 -> 概念之间的作用关系 -> 首个可观察变化 -> 现象/结果
```

只保留支撑主线的信息：

- *问题*：具体锚点、旧做法怎么判断、它在哪里失灵。
- *模型*：哪些概念在什么条件下怎样作用，为什么会生成眼前现象。
- *证据*：足以校准模型中承重关系的结果；通常只需 1-2 个，关系较多时以覆盖为准，不以数量为准。
- *分寸*：证据最多把模型推到哪里，哪些关系仍只是作者解释或结构推演。
- *启发*：以后遇到什么，我多问哪一个问题或改哪一个动作。
- *元数据*：标题、作者、来源、年份。

模型过三道闸：

1. *解释原现象*：把模型放回具体锚点，能说明为何出现该结果，而不是换一组词复述结果。
2. *区分邻近情况*：选择一个表面相似、但关键概念或关系不同的案例，模型应预测不同结果。
3. *预测条件变化*：让一个关键概念变强、变弱或缺席，写出最先变化的可观察信号及其如何传到最终结果。

模型不是论文已经证明的代名词。每条承重关系都要有相邻证据，或明确标为作者解释 / 结构推演；同一个证据可以同时约束多条关系。完整实验流程、全量指标、所有 baseline、公式推导、相关工作和实现参数默认不写；只有它会改变模型或证据边界时才进入正文。

## 第七步：扫描关系阻力与证据阻力

完成概念与解释模型后，问两个问题：

1. *关系阻力*：概念之间的作用是否难以仅靠文字在脑中成形？
2. *证据阻力*：实验结果与中心主张之间的关系是否容易被读者高估或误读？

为最大的瓶颈选择成本最低的表征：

| 论文中的思想形状 | 优先表征 |
|---|---|
| 模型架构、数据流、模块关系 | 结构图或流程图 |
| 因果链、训练过程、状态变化 | 因果图或时间轴 |
| 反馈、迭代更新、自我强化 | 回路图 |
| 分布、尺度效应、边际变化 | ASCII 曲线 |
| 准确率与成本、速度与质量 | trade-off 曲线 |
| 旧指标与新指标的差异 | 小表格或二维矩阵 |
| 一个案例如何逐步变化 | 对齐的 example trace |
| 公式本身就是核心机制 | 一个最小公式，加人话翻译 |
| 两三句话已经足够 | 纯文字 |

图不是标配。若使用图，把它放在概念已经用人话和案例落地之后；图负责让关系显形，不替代概念形成。让相关证据与适用边界紧随其后。ASCII 图放进 `#+begin_example` / `#+end_example`，宽度不超过 80 字符。

只看图，读者应能复述「哪些概念怎样作用、模型怎样生成现象、它改变了什么判断」，同时不会把局部实验误解成普遍规律。若图只是把摘要换行，或掩盖条件、代价与未证明部分，删除它。

## 第八步：按理解顺序写

写作顺序是「问题 → 洞见 → 带走」，最后压出「速读」。

### 速读

用一个短段落交代问题、洞见、带走。不设三个小标题，也不附泛泛免责声明。允许压缩正文一次，不逐句复制。

### 它到底在解决什么

从具体锚点开场，接着说明人们原来怎么处理、为什么在这里失灵。讲到读者开始期待新答案就停，不提前展开方法，也不虚构领域共识。

### 它真正看见了什么

先用一句人话写新判断。随后沿模型的关系链推进：某个概念只有在下一步关系需要它时才出现，并立刻回到同一锚点，让读者看到它区分什么、怎样起作用。全文要覆盖账本中的全部承重缺口，但不要逐概念写卡片或固定段落。已经熟悉或不承重的词不凑数；若确实没有新概念，明确指出论文改变的是旧概念之间的关系，而不是发明术语。

沿同一锚点运行完整的最小模型，并用一个邻近情况或条件变化展示它的区别性预测。若第七步发现文字不足，就在这里加入选定的图、表、trace 或最小公式。证据紧跟它支持的关系；通常只留 1-2 个改变判断的结果，但每条没有直接证据的承重关系都要明确标出其性质。

用动词校准分量：

- 多模型、多任务且有反例控制：「较稳地显示」。
- 范围窄但现象清楚：「在这些条件下显示」。
- 单点结果或早期观察：「提示」「值得怀疑」。
- 评测或数据集：「把……变得可测」「暴露旧指标的盲点」。

适用范围也就地写：「这最多说明……」「它还没有说明……」「这只在……条件下成立。」没有真实误用风险就不强行加。

### 我能带走什么

只写一条可复用的判断规则：

> 以后看到 {现象}，我先不急着判断 {旧结论}，而会先问 {新问题}。

必要时补一到两个真正相关的用法。没有直接用途，就说明它更像识别问题的尺子，不硬凑产品、研究和人生三套启发。

## 写作约束

- 用中文日常表达。术语先说人话，再括号给原词。
- 不把「概念」写成术语翻译。单个承重概念必须出现区别、关系和案例；这些概念接成的模型能预测条件变化时，理解才真正闭合。
- 概念缺口要完整，概念表述要克制：宁可少收外围词，也不漏掉让中心命题和证据链成立的词。
- 不用学术套话开场，不写「本文提出」「实验表明该方法优于 baseline」。
- 类比只在能承载机制时使用，不做装饰。
- 数字最多保留 1-2 个，并翻译成它改变了什么判断。
- 主文不堆 LaTeX 公式。若一个关系式就是核心洞见的最短表达，可以保留一个最小公式，但必须立刻翻译成自然语言，并说明它改变了什么判断。
- 中心判断只完整展开一次；速读可以再压缩一次。
- title 是 6-18 个中文字符的判断，不是题目翻译或方法名；看不出方向时用中文 subtitle 兜底。

## 自查

- 只读速读，能否说清问题、洞见、带走？
- 全文是否围绕同一个具体锚点？
- 是否扫描了中心命题、关系和证据链，记录全部承重的概念缺口？
- 每个概念是否在区别、关系和案例中工作，而不只是定义或类比？
- 概念是否按依赖顺序连接，避免变成按论文顺序罗列的术语表？
- 模型是否解释原锚点、区分一个邻近情况，并预测一个条件变化？
- 洞见是否符合论文类型，没有把评测、工具或数据集拔高？
- 证据和适用范围是否紧跟各自修饰的主张？
- 是否扫描了关系阻力与证据阻力，并选了成本最低的表征？
- 解释模型若具有结构、形状或动态变化，是否真正显形，而非只用文字带过？
- 图表若存在，是否既能复述机制，又不会让结论越过证据边界？
- 若保留最小公式，是否紧接人话翻译，而非把公式本身当作解释？
- 启发是否落成一个问题或动作，而不是场景清单？
- 删掉章节标题，正文是否仍像一个人在解释同一件事？
