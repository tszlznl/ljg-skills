# Reading Guide：建立论文的 x → f → f(x)

这份指南把论文从学术写作顺序改写成外行可理解的研究逻辑。先读完整材料，在内部接通 x、f、f(x)，再开始写。

## 1. 先判材料与论文类型

优先使用原始 PDF、HTML 全文、正式论文页或作者版本。只有标题时先找到可靠原文；遇到出版墙，转向 arXiv、作者主页或机构仓储，不用二手摘要补正文。

按材料标记：

- *完整解读*：有全文及关键图表、方法和实验。
- *初读*：只有摘要、扩展摘要、演讲或部分正文。
- *假设版*：材料很薄，只能提出暂定理解。

再判一个主类型：

| 类型 | f 通常是什么 | f(x) 主要看什么 |
|---|---|---|
| 解释 / 发现 | 新关系或因果机制 | 现象能否被更好解释 |
| 方法 / 干预 | 算法、流程或训练信号 | 目标是否改善、代价是否变化 |
| 测量 / 评测 | 新指标、任务或判定维度 | 旧尺子的盲点是否暴露 |
| 资源 / 系统 | 数据、工具或基础设施 | 原本做不到的事是否变得可行 |
| 综述 / 理论 | 组织事实的框架或形式关系 | 分散事实是否获得统一解释 |

类型限制主张分量。评测论文通常改变「怎么量」，资源论文通常改变「能不能做」，不能自动拔高成世界运行机制。

## 2. 定义 x：研究问题与旧方案缺口

x 不是领域重要性介绍，而是一个旧方法在具体条件下失败的问题。

写清四件事：

- 对象：谁或什么系统遇到问题。
- 目标：原本想完成什么。
- 旧路：baseline 或惯常做法怎样处理。
- 缺口：它在哪个可观察场景里失败。

找一个具体锚点贯穿全文，例如：模型答案已经正确，却继续生成三倍长度的推理；proposal 主题相近，却接错父论文的核心机制。

若论文是首创工作，没有明显 baseline，不硬造对手；写清此前为何不可测、不可做或无人处理。

## 3. 提炼 f：论文改了哪一步

f 必须是一句会运转的机制：

> 旧方法根据 A 做 B；论文额外利用 C，把 B 改成 D，因此可能缓解 x。

不同论文的 f：

- 方法论文：输入、关键中间信号、决策规则、输出。
- 解释论文：变量关系、作用方向、产生现象的机制。
- 评测论文：测量对象、旧指标盲点、新增维度、判定结果。
- 资源论文：原有瓶颈、资源提供方式、可行性边界怎样移动。
- 理论论文：核心对象、约束、推出的关系；公式要翻成自然语言。

只保留 2 到 4 个不能删的部件。组件多但关系没讲清，仍然不算理解方法。

## 4. 用 ASCII 图呈现方法或比较

先决定读者最需要看清什么：

### 方法流

适合算法、系统、训练流程和干预论文：

```org
#+begin_example
输入 ──> 旧流程 ──> 失败点

输入 ──> 新信号 ──> 新决策 ──> 输出
#+end_example
```

### 方案对比

适合论文关键贡献是「换了什么信息或判断标准」：

```org
#+begin_example
              使用的信息       关键动作       主要盲点
旧方案 A      ...              ...            ...
旧方案 B      ...              ...            ...
论文 f        ...              ...            ...
#+end_example
```

### 因果或测量关系

适合解释、理论和 benchmark 论文：

```org
#+begin_example
旧尺子：可见信号 ─────────────> 结论
                    └─ 漏掉：机制关系

新尺子：可见信号 + 机制关系 ──> 新结论
#+end_example
```

每篇通常只选一种。图宽不超过 80 字符；baseline 最多 2 到 3 个；只比较能解释 f 的维度。图后用一小段指出读图结论，不再逐行复述。

## 5. 计算 f(x)：结果与证据边界

把论文的 f 放回 x，回答：问题改善了吗，凭什么这样说？

选择 2 到 4 个关键证据：

- 最能挂回 x 的主结果。
- 一个说明机制确实工作的消融或对照。
- 一个暴露代价、失败条件或范围边界的结果。
- 必要时，一个最有解释力的反直觉发现。

数字通常保留 1 到 3 个。不要只写「提升显著」，要翻译成它改变了什么判断。不要把 benchmark 内结果直接推广到现实世界。

用词校准分量：

- 多模型、多任务且有反例控制：「较稳地显示」。
- 范围窄但现象清楚：「在这些条件下显示」。
- 单点结果或早期观察：「提示」「值得怀疑」。
- 测量或资源贡献：「使……可测」「使……变得可行」。

## 6. 从结果推出判断更新

f(x) 的最后一层是读者该改判什么，不是额外应用清单。

可以落在：

- 以后评价同类方法，先看哪个机制或假设。
- 以后遇到同类失败，先诊断能力不足还是反馈缺失。
- 设计系统时，调整一个决策顺序或测量维度。
- 暂时不更新行动，只降低或提高对某个主张的置信度。

若判断拿掉 f 后仍像普遍正确的话，说明与论文脱节。没有直接用途时，诚实写它只更新研究判断。

## 7. 写正文

正文通常 1000 到 1800 个汉字，不含文件头与资料校准。按理解顺序写，不按论文章节顺序复述。

### x：论文要解决什么问题

从具体失败开场，说明旧方案和缺口。讲到读者开始期待新解法就停。

### f：作者提出了什么解法

先用一句人话说清 f，再解释最小机制。方法流或 baseline 对比 ASCII 图放在本节；图后点明关键差异。

### f(x)：结果改变了什么判断

先写论文内结果与适用条件，再写由这些结果直接推出的判断更新。证据贴着它支持的主张。

### 资料校准

列 2 到 5 条：论文原文、代码或数据（若使用）、必要外部校准、材料等级。每条只写来源及其用途，不补文献综述。

## 8. 输出格式

保存到：

`~/Documents/notes/{时间戳}--paper-{方法名或论文关键词}__paper.org`

文件头：

```org
#+title:      {一句中文判断，优先写 f(x) 的判断更新}
#+subtitle:   {论文原始标题；必要时加中文解释}
#+date:       [{YYYY-MM-DD Day HH:MM}]
#+filetags:   :paper:
#+identifier: {YYYYMMDDTHHMMSS}
#+source:     {原始 URL 或来源}
#+authors:    {作者列表}
#+venue:      {发表场所/年份}
```

## 9. 文风

- 用中文日常表达；术语先说人话，再括号给原词。
- 不用学术套话开场，不写「本文提出」「实验表明优于 baseline」。
- 主文不放 LaTeX；必要公式翻成变量关系。
- 不做正式同行评审，不给 accept / reject 判决。
- 不写来源、承重、贡献类型、挪动类型等元分析标签。
- title 是 6 到 18 个中文字符的判断，不是论文标题翻译。

## 10. 交稿前检查

依次补完：

> 这篇论文试图解决 ______。

> 它把旧方法的 ______ 改成 ______，因为 ______。

> 在 ______ 条件下，结果显示 ______，所以我会改判 ______。

第二句若只剩方法名，f 没展开；第三句若没有条件和证据，f(x) 在过度外推；第三句的判断若删掉论文仍成立，就还没真正带走这篇论文。
